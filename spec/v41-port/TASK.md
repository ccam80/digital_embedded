# v41 Port — Applier Contract

This is the task contract for the ngspice **v26 → v41** port loop. It governs
the agent that *applies* changes. The agent that *verifies* them works from
`VERIFICATION.md`. The two are never the same context.

---

## 1. Mission

`ref/ngspice` has been moved from a v26-era tree to the `ngspice-41` tree. Our
TypeScript engine and SPICE primitive models are a **line-for-line, byte-equal
port** of the v26 ngspice C source. They must become a line-for-line,
byte-equal port of **v41**.

The diff archives under `spec/ngspice-v41-model-diffs/` and
`spec/ngspice-v41-engine-diffs/` are the complete, unfiltered upstream
`v26 → v41` `git diff`. Your job is to apply that delta to our TypeScript so
that, **above the level of an identifier rename and C↔TS syntax**, our code is
identical to v41 ngspice:

- arithmetic operations identical and identically **ordered**;
- control flow identical;
- grouping into functions identical (a v41 function split → a method split);
- statement order identical.

## 2. The one rule that overrides everything

> **"This doesn't fit our architecture" is never a disposition. It is the
> task.** If a v41 change does not fit our current architecture, you change
> our architecture so that it does. Updating the architecture to match v41
> *is the work* — it is not a reason to skip, soften, reinterpret, or defer a
> hunk.

Two prior attempts failed because agents judged diff chunks for "salience" and
dropped the ones that did not fit. There is no salience judgment in this task.
Every hunk is applied or formally accounted for. Nothing is dropped.

## 3. Total enumeration — the ledger

`ledger.json` (regenerated by `build-ledger.mjs`) contains **one item per
git-diff hunk** across every diff doc — 100% of the diff, machine-counted. The
loop is complete only when **zero items are `PENDING`**.

Every item ends in exactly one state:

| State | Meaning | Who may set it |
|---|---|---|
| `APPLIED` | TS delta is structurally isomorphic to the v41 hunk; verifier confirmed. | Verifier only |
| `ESCALATED` | Blocked on a user decision (§8). **Not done** — blocks job completion. | Applier or verifier |
| `NO-COUNTERPART` | The ngspice hunk has no behavioral counterpart in our port. | **Phase 0 planning only — frozen.** |

`NO-COUNTERPART` is assigned **before the loop starts**, by the planning pass,
per ngspice file, and reviewed by the user. **You — the loop applier — can
never assign `NO-COUNTERPART`.** If you believe an item should be
`NO-COUNTERPART` and it is not already marked so, that is an **escalation**,
not a decision you make.

There is no "skipped", "deferred", "out of scope", or "not applicable" state.
Those are not representable. An item you do not apply stays `PENDING` and the
loop cannot terminate — by construction.

A ralph **run** ends when no item is `PENDING`. The **job** is done only when
no item is `PENDING` **and** no item is `ESCALATED` — every item is `APPLIED`
or (Phase-0-frozen) `NO-COUNTERPART`. `ESCALATED` is a *blocking* state, not a
terminal one: a run that ends with escalations open reports them and stops;
the user resolves them and the loop is re-run. Escalation can never make the
job "complete".

## 4. Unit of work

Work **one `functionGroup` at a time** (the `functionGroup` field groups every
hunk in one ngspice function). Within a group, apply hunks in **source order**
(ascending `oldStart`) — a later hunk often assumes an earlier one landed.

Pick the next group whose file's `classification` is `PORT` or `REVIEW-CLASS`
and that has `PENDING` items, processing diff docs in the order set by the
run plan (pilot device first; small models before large; engine last).

Files classified `NO-COUNTERPART` are already fully dispositioned — skip them;
they are not `PENDING`.

A ledger item may be a **sub-item** of a split hunk — its `id` carries a letter
suffix (`#h003a`), `isSubItem` is `true`, and `docLineRange` covers only its
slice of the parent hunk. A sub-item behaves exactly as a hunk: apply it
against its own `docLineRange`. Sub-items share their parent's `functionGroup`,
so they group for acceptance like any other hunk in that function.

## 5. What `APPLIED` means — the diff-of-diffs standard

### The disposition rule — APPLY or ESCALATE, nothing else

For every hunk you have exactly **two** possible outcomes: **APPLY** it, or
**ESCALATE** it (§6, §8). There is no third outcome — no "no-counterpart", no
"already equivalent", no "cosmetic, nothing to do". You may **APPLY** a hunk
only if **all four** of these hold; if any fails, or you are unsure whether it
holds, you **ESCALATE**:

1. **Pre-image present** — every v26 `-` line is present in our current TS,
   modulo identifier rename and C↔TS syntax (§6 case (a)).
2. **Target present** — the construct the v41 `+` lines modify or extend
   exists in our TS, or can be created by an architecture change **contained
   within this `functionGroup`**.
3. **Isomorphic delta** — the change renders as a `git diff` of our TS that is
   line-isomorphic to the ngspice hunk (below).
4. **No ambiguity** — the ngspice→digiTS mapping is not genuinely ambiguous.

"Our architecture already achieves this by other means", "digiTS is simpler
here", "this is cosmetic for us", "the post-change behaviour is already
present" are **not** APPLY conditions and **not** escape routes. If a hunk's
`+`/`-` content has no line-isomorphic image in our code, that is condition 1
or 2 failing → **ESCALATE**. You never decide a hunk has no counterpart — only
frozen Phase-0 planning does that (§3).

### Applying the delta

Do **not** try to make our whole file equal a C file. Apply the **delta**:

1. The hunk's `-` lines are **v26** ngspice; its `+` lines are **v41**.
2. Edit our TS so that `git diff` of your edit is **line-isomorphic** to the
   ngspice hunk: the same lines removed, the same lines added, in the same
   positions and order — modulo identifier rename and C↔TS syntax only.

A **zero-line TS delta** is a valid application of a hunk **only if every `+`
and `-` line of the hunk is itself an allowed difference** (identifier rename,
C↔TS syntax, or comment). If any `+`/`-` line is a real change — a new
statement, a new branch, changed arithmetic, changed control flow — a
zero-line delta has **dropped** it: that is condition 3 failing → ESCALATE. It
is never a clean apply.

**Allowed differences — this list is closed.** A difference you cannot place
in one of these three classes is forbidden; if you are unsure which it is,
ESCALATE:

- identifier names (tracked in the rename map, §7);
- C↔TS syntax: `->` vs `.`, pointer deref, `double`/`number`, typed-array vs
  C-array indexing, brace style, and the surface spelling of a `for`/`while`/
  `if` whose condition, body, and structure are otherwise identical;
- comment wording.

**Forbidden** — these mean the port is wrong, not "equivalent":

- reordering operands of an arithmetic expression;
- factoring out or inlining a subexpression;
- merging or splitting statements;
- algebraic simplification (`a*b + a*c` → `a*(b+c)`);
- changing branch/loop nesting, or adding/removing a branch;
- introducing a helper ngspice does not have, or merging functions ngspice
  keeps separate.

If v41 adds a function, you add the method. If v41 deletes one, you delete it.
If v41 moves code between functions, you move it the same way.

## 6. Pre-image check — mechanical, no judgement

Before editing, classify the hunk's v26 `-` lines against our current TS.
This is a **lookup, not an assessment**. For each `-` line, exactly one holds:

- **(a) present** — a TS line *is* that `-` line, modulo identifier rename and
  C↔TS syntax.
- **(b) present but differs** — a TS line is clearly its counterpart but is
  not that `-` line modulo rename/syntax. Our port already diverged from v26.
- **(c) absent** — no TS line is that `-` line; the construct it belongs to
  does not exist in our code at all.

APPLY (clean delta) when **every** `-` line is case (a). When a `-` line is
(b) or (c), the v26 baseline this hunk assumed is not what our code holds — but
**the target is v41, not v26**, and the diff is only the guide to *what changed*.
Your disposition:

- **Bad baseline, fixable in-group → PORT THE WHOLE FUNCTION TO v41.** If you can
  make our function bijectively match the fresh **v41** source
  (`ref/ngspice/<file>`) with edits **contained within this `functionGroup`**, do
  that directly instead of applying a delta onto a baseline we do not have. You
  are *not* "restoring v26 then patching" — you write the function to v41. This is
  the applier's job (§2), **not** an escalation. Record it for audit: add a
  `verifierNotes` line on each affected hunk's progress entry beginning
  `BASELINE-DIVERGENCE:` that names the function and what diverged from v26, so the
  drift stays visible without a blocking round-trip.
- **Sequencing block:** if the hunk carries `blockedBy` and that reconstruction
  item is not yet `APPLIED`, do **not** work it — skip, touch nothing, record
  nothing (see "Blocked hunks" below). Not an escalation.
- **Out-of-group / feature gap → ESCALATE.** Only when matching v41 requires a
  change *beyond this `functionGroup`* (other files, shared infra) or a feature
  digiTS's model genuinely lacks. There you cannot mechanically tell a bug from an
  intentional divergence — it is the user's call.

**Already at v41.** If a `-` line is absent because our code is *past* v26 (the
function was written straight at v41, or this is a `STALE` re-verify), that is
**not** "already achieves, skip". Confirm the whole function bijectively matches
the fresh v41 source and report the group ready; the verifier records `APPLIED`
with a note that the v26 pre-image is absent. The `APPLIED` is earned by the v41
bijective match — **never** by the mere absence of a delta.

You may **never** reclassify a `PENDING` hunk as `NO-COUNTERPART` (frozen Phase-0
only, §3), and you may never treat an absent pre-image as "plumbing, so fine".

A hunk that is **all `+` lines, no `-` lines** (adds a wholly new construct)
has a trivially-satisfied pre-image; for it, §5 condition 2 (target present)
is the gate — if there is no digiTS construct to extend and creating one
exceeds this `functionGroup`, ESCALATE.

**Blocked hunks.** A ledger item may carry `blockedBy` — the id of a
v26-baseline reconstruction item (the planner found digiTS never ported the
function this hunk builds on). If `blockedBy` is set and that reconstruction
item is not yet `APPLIED`, **do not work this hunk** — skip it, touch nothing,
record nothing. It is not an escalation; it is a sequencing block. The hunk
becomes workable once its reconstruction item is `APPLIED`, at which point its
pre-image will be case (a). Reconstruction items are built from their own spec
in a pre-phase — they are not applied by this diff-of-diffs loop. A case-(c)
absent pre-image on a hunk with **no** `blockedBy` is a planning miss —
ESCALATE as above.

Your escalation names: the ngspice file + hunk id; the v26 `-` lines and the
actual current TS (or its absence); the specific divergence.

## 7. Rename map

`rename-maps/<device>.md` records the ngspice-identifier → digiTS-identifier
mapping for the device. A stub is pre-created for you — maintain it by `Edit`
(never create it yourself): every identifier you map, add a row. It is
**documentation**. The verifier does **not** consume it — the verifier
re-derives equivalence independently, so a wrong map cannot produce a false
`APPLIED`. Keep it accurate anyway: it keeps you consistent across the
device's hunks.

## 8. Escalation — a pause, never an exit

This whole job **is** the architectural alignment. There is no "accepted
divergence" outcome and no file that records one. Every behavioral difference
from v41 ngspice is either not-yet-done (`PENDING`), done (`APPLIED`), or
nonexistent-by-design (`NO-COUNTERPART` — ngspice code with no behavioral
counterpart in our port, Phase-0 frozen). Escalation never produces a fourth
outcome. An escalated item is **not done**: it blocks job completion until a
user decision drives it back to `PENDING` and then to `APPLIED`. You gain
nothing by escalating except a round trip and visibility.

> Your `CLAUDE.md` mentions `spec/architectural-alignment.md` as the
> project-wide home for accepted divergences. **That route does not apply to
> this job.** This job's purpose is to drive divergences to zero, so there is
> no such file and no such outcome here. Escalations go to `ESCALATIONS.md`,
> are reviewed by the verifier, and block job completion.

You **ESCALATE whenever §5's four APPLY conditions are not all met.**
Escalation is the **default** outcome for any hunk you cannot cleanly apply —
not a rare exception. It is far better to escalate a hunk that turns out
applicable (the verifier or the user bounces it straight back) than to force a
hunk you do not understand into a false APPLY. Over-escalation is visible and
self-correcting; a false APPLY is silent and corrupts the ledger.

The common shapes of escalation — not a restrictive list:

1. **Bad baseline needing an out-of-group fix (§6)** — a v26 `-` line differs
   from or is absent from our code AND bringing the function to v41 needs a
   change beyond this `functionGroup` (or a feature digiTS's model lacks). A bad
   baseline you *can* fix in-group is **not** an escalation — you port the whole
   function to v41 per §6 and record a `BASELINE-DIVERGENCE:` note.
2. **Target absent / cross-group** — the v41 `+` lines have no digiTS
   construct to land in, or building one needs an architecture change beyond
   this `functionGroup` (other files, other groups, shared infrastructure).
3. **Genuine C→TS ambiguity** — the mapping is ambiguous in a way no rename or
   syntax rule resolves.
4. **You are unsure** — any case where you cannot positively confirm all four
   §5 APPLY conditions. Uncertainty is an escalation, not a coin-flip.

ESCALATE is **not** for a hunk that genuinely meets all four §5 conditions —
apply those, including any architecture change contained within the
`functionGroup` (§2). The bar is not on *whether* to escalate (escalate
freely) — it is on the **quality of the escalation report**: it must be
specific (below), so the user or verifier can act on it.

The bar for a valid escalation is high on purpose: name the cited ngspice
file + lines, the digiTS file, the specific architecture change required, and
every file it touches. If you can describe the change that precisely, you can
usually just make it — so a well-formed escalation is rare. A vague escalation
is bounced straight back to `PENDING` (by the verifier or the user) with the
directive "make it match ngspice", because that is nearly always the answer.

To escalate: write the item an `ESCALATED` entry in
`spec/v41-port/progress.json` (state `"ESCALATED"`, the `escalation` object,
the item's current `hunkHash`) — see §10 — append the same to
`ESCALATIONS.md`, then set that group aside and continue with other groups.
The loop keeps moving; the **job** does not complete while any item is
`ESCALATED`.

## 9. Per-iteration procedure

1. Read `ledger.json`; pick the next `functionGroup` with `PENDING` items.
2. For each hunk in the group, in `oldStart` order:
   a. `Read` the hunk from its `diffDoc` at `docLineRange`.
   b. `Read` the v41 ngspice function from `ref/ngspice/<ngspiceFile>` for
      surrounding context — never rely on the diff markdown alone.
   c. `Read` our TS function (`files[].tsFile`, `tsFunction`).
   d. Pre-image check (§6). On failure → escalate, stop the group.
   e. Apply the delta (§5). Update the rename map (§7).
3. Self-review the whole group against the v41 function: arithmetic, order,
   control flow, function boundaries.
4. On a clean apply, write **nothing** to `progress.json` — leave the items
   `PENDING` and report the group **ready for verification**. The verifier —
   not you — records `APPLIED`. You touch `progress.json` only to escalate
   (§8) or to bump `attempts` after a rejection (§10).
5. Commit nothing. The verifier and the loop driver handle state.

## 10. State protocol — the progress overlay

You never edit `ledger.json` — it is machine-derived. Loop state lives in
`spec/v41-port/progress.json`, an overlay keyed by item id. You `Edit` that
small file; `build-ledger.mjs` composes it into `ledger.json`. After writing,
run `node spec/v41-port/build-ledger.mjs` — it validates your entry and fails
loudly on a malformed one.

You may write a progress entry that sets `attempts` (increment when you
re-attempt a verifier-rejected item) and, to escalate, `state = "ESCALATED"`
with the `escalation` object. Every entry carries the item's current
`hunkHash`. Append to `verifierNotes` only to record your response to a prior
rejection.

You may **not** write `state = "APPLIED"` (the verifier's) and you may not
touch a `NO-COUNTERPART` item (Phase-0 frozen — `build-ledger.mjs` rejects a
progress entry on one). Recording your own work `APPLIED` is the exact failure
this contract exists to prevent.

When a verifier rejection leaves an item `PENDING` with a note, read the note,
fix the **specific** divergence it names, and increment `attempts` in the
item's progress entry. Do not re-submit the same edit. At `attempts >= 3` the
item auto-escalates — if you are about to make a third attempt, escalate
instead with what you have tried and why it keeps failing.

## 11. Tests are not your gate

Some tests will break and some will start passing — expected. **Do not** edit
code to make a test pass, do not weaken an assertion, do not add a skip/xfail.
Your gate is structural identity to v41 (§5), verified per `VERIFICATION.md`.
Test triage is a separate final phase, arbitrated by the v41 harness — not by
you, and not during the loop.

## 12. Banned closing verdicts

Per `CLAUDE.md`, these words are banned as a verdict on any divergence:
*mapping*, *tolerance* / *within tolerance* / *close enough*, *equivalent to*,
*pre-existing*, *intentional divergence*, *partial*. If you reach for one,
STOP and escalate instead.
